= Authenticate to OAuth2 and OIDC services
:page-role: howto
include::includes/attributes.adoc[]

When a workflow calls a protected HTTP or OpenAPI service, it usually needs an `Authorization: Bearer <token>` header. Instead of fetching and refreshing those tokens yourself, Quarkus Flow can negotiate them for you: you declare an OAuth2/OIDC authentication on the call, and the engine performs the token request against the authorization server, caches the result, and attaches it to the downstream request automatically.

This guide shows how to:

* Enable OAuth2/OIDC token negotiation in a Quarkus Flow application.
* Attach a token to an HTTP or OpenAPI call using the most common grants (Client Credentials, OIDC, Password, Token Exchange).
* Reuse a single authentication across several calls, and use multiple authorization servers in one workflow.
* Keep client credentials out of your workflow code using xref:secrets.adoc[secrets].

[TIP]
A complete, runnable project covering every scenario below lives in `examples/auth-oauth2-oidc`. All authorization servers and downstream APIs there are mocked with https://docs.quarkiverse.io/quarkus-wiremock/dev/[WireMock Dev Services], so it runs end-to-end without a real OAuth2 server.

== Prerequisites

* A Quarkus application with xref:getting-started.adoc[Quarkus Flow set up].
* Familiarity with xref:http-openapi-tasks.adoc[HTTP and OpenAPI tasks] and xref:secrets.adoc[secrets].

== 1. Add the OIDC client dependency

Add `quarkus-oidc-client` to your project. Quarkus Flow's OIDC integration activates automatically when this dependency is on the classpath:

[source,xml]
----
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-oidc-client</artifactId>
</dependency>
----

You declare which grant and credentials to use on the call; Quarkus Flow obtains the token, caches it, refreshes it when needed, and adds the `Authorization` header. To opt out, set `quarkus.flow.oidc.enabled=false` (see the <<configuration-reference>>).

.When is a token cached?
[%collapsible]
====
Quarkus Flow creates an `OidcClient` lazily on the first call that needs it and keeps it for the lifetime of the application. Every subsequent call whose resolved policy matches the same cache key reuses that client — and therefore its token cache and automatic refresh logic.

The cache key covers: authority, token endpoint path, OAuth2-vs-OIDC flag, client ID, client secret, client-authentication method, grant type, scopes, audiences, and username/password (for the Password grant).

Two call sites share a cached client **only when every field matches**. If any field differs — even a single scope — a separate `OidcClient` is created with its own independent token lifecycle:

* **Same authority, same client, same grant, same scopes** → one cached client, one token.
* **Same authority, same client, different scopes** → two cached clients, two independent tokens.
* **Same authority, different token endpoint paths** → two cached clients, even if everything else matches.
* **`oauth2(...)` vs `oidc(...)` for the same authority** → two cached clients, because the OAuth2-vs-OIDC flag differs.

All cached clients are closed automatically on application shutdown.
====

== 2. Attach an authentication to a call

You can attach an authentication **inline** on a single call, or declare it **once** under `use(...)` and reference it by name from many calls.

=== Inline on a single call

[source,java]
----
import static io.serverlessworkflow.fluent.func.dsl.FuncDSL.*;

http("listImages")
    .GET()
    .header("Accept", "application/json")
    .uri(URI.create(imageService),
         oauth2(baseUrl, CLIENT_CREDENTIALS, clientId, clientSecret)); // <1>
----
<1> `oauth2(...)` attaches the authentication directly to this call. Use `oidc(...)` for an OpenID Connect client.

=== Declare once under `use(...)` and reference by name

If several calls share the same authority, declare the authentication once and reference it by name:

[source,java]
----
workflow("orders")
    .use(use -> use.authentications(auth ->
        auth.authentication("inventory", a -> a.oauth2(o -> o
            .authority(baseUrl)
            .grant(CLIENT_CREDENTIALS)
            .client(c -> c.id(clientId).secret(clientSecret))))))
    .tasks(
        http("reserve").GET()
            .uri(URI.create(inventory + "/reserve"), use("inventory")), // <1>
        http("confirm").GET()
            .uri(URI.create(inventory + "/confirm"), use("inventory")))
    .build();
----
<1> `use("inventory")` references the named authentication declared above.

== 3. Use the Client Credentials grant

If you need machine-to-machine authentication, use the Client Credentials grant. Read credentials from xref:secrets.adoc[secrets] with `${ $secret.* }` expressions rather than hardcoding them:

[source,java]
----
@ApplicationScoped
public class ClientCredentialsFlow extends Flow {

    @ConfigProperty(name = "image.service.url")
    String imageService;

    @ConfigProperty(name = "clientCredentials.baseUrl")
    String baseUrl;

    @Override
    public Workflow descriptor() {
        return FuncWorkflowBuilder.workflow("client-credentials", "quarkus-flow")
                .use(u -> u.secrets("clientCredentials")) // <1>
                .tasks(
                        FuncDSL.call(
                                FuncDSL.http("listImages")
                                        .GET()
                                        .header("Accept", "application/json")
                                        .uri(URI.create(imageService),
                                                FuncDSL.oauth2(baseUrl, CLIENT_CREDENTIALS,
                                                        "${ $secret.clientCredentials.clientId }",
                                                        "${ $secret.clientCredentials.clientSecret }")))) // <2>
                .build();
    }
}
----
<1> Declares the `clientCredentials` secret group, making its values available to the workflow.
<2> The id and secret are resolved from that group at runtime.

The secret values must be supplied externally. For local development and tests, `application.properties` is the simplest option:

[source,properties]
.application.properties
----
clientCredentials.clientId=quarkus-flow
clientCredentials.clientSecret=dummy-client-secret
clientCredentials.baseUrl=https://auth.example.com/realms/test/protocol/openid-connect/token
----

In production, the same secret names resolve through any Quarkus https://quarkus.io/guides/credentials-provider[`CredentialsProvider`] (HashiCorp Vault, Kubernetes Secrets, …) without changing the workflow code. See xref:secrets.adoc[Resolve secrets securely] for details.

== 4. Use an OIDC client

If the authorization server is an OpenID Connect provider, use `oidc(...)` instead of `oauth2(...)`:

[source,java]
----
FuncDSL.http("listImages")
        .GET()
        .header("Accept", "application/json")
        .uri(URI.create(imageService),
                FuncDSL.oidc(baseUrl, CLIENT_CREDENTIALS,
                        "${ $secret.oidcClient.\"client-id\" }",      // <1>
                        "${ $secret.oidcClient.\"client-secret\" }"));
----
<1> When a secret key contains a dash, quote it inside the JQ path: `${ $secret.oidcClient."client-id" }`.

== 5. Use the Password grant

If the service requires a username and password alongside client credentials, use the full `oauth2(...)` builder:

[source,java]
----
FuncDSL.http()
        .method("DELETE")
        .uri(URI.create(imageService + "/attrs/" + id),
                FuncDSL.oauth2(oauth2 -> oauth2
                        .authority(baseUrl)
                        .grant(PASSWORD)
                        .client(c -> c.id("${ $secret.\"password-grant\".\"client-id\" }")
                                      .secret("${ $secret.\"password-grant\".\"client-secret\" }"))
                        .username("${ $secret.\"password-grant\".username }")
                        .password("${ $secret.\"password-grant\".password }")));
----

== 6. Use Token Exchange (RFC 8693)

If you need to exchange a subject (and optional actor) token for a new one scoped to a target audience, use the token-exchange grant:

[source,java]
----
FuncDSL.oauth2(oauth2 -> oauth2
        .endpoints(token -> token.token("/oauth2/token"))
        .client(c -> c.id("my-client").secret("my-secret"))
        .subject(s -> s.token("${ $secret.exchangeSecrets.subjectToken }")
                       .type("urn:ietf:params:oauth:token-type:access_token"))
        .actor(a -> a.token("${ $secret.exchangeSecrets.actorToken }")
                     .type("urn:ietf:params:oauth:token-type:access_token"))
        .scopes("api")
        .audiences("target-service")
        .grant(URN_IETF_PARAMS_OAUTH_GRANT_TYPE_TOKEN_EXCHANGE)
        .authority(baseUrl));
----

== 7. Call multiple authorization servers

If your workflow calls services protected by different authorization servers, declare each authentication under `use(...)` and reference them by name:

[source,java]
----
return FuncWorkflowBuilder.workflow("multiple-oauth2-clients", "quarkus-flow")
        .use(use -> use.secrets("joogle", "jahoo") // <1>
                .authentications(auth -> {
                    auth.authentication("joogle", a -> a.oauth2(o -> o
                            .endpoints(e -> e.token("/auth/joogle/token"))
                            .client(c -> c.id("${ $secret.joogle.clientId }")
                                          .secret("${ $secret.joogle.clientSecret }")
                                          .authentication(CLIENT_SECRET_POST))
                            .authority(joogleBaseUrl)
                            .grant(CLIENT_CREDENTIALS)));
                    auth.authentication("jahoo", a -> a.oauth2(o -> o
                            .endpoints(e -> e.token("/auth/jahoo/oidc/token"))
                            .client(c -> c.id("${ $secret.jahoo.clientId }")
                                          .secret("${ $secret.jahoo.clientSecret }")
                                          .authentication(CLIENT_SECRET_POST))
                            .authority(jahooBaseUrl)
                            .grant(CLIENT_CREDENTIALS)));
                }))
        .tasks(
                FuncDSL.fork( // <2>
                        FuncDSL.http("getEmailsFromJoogle").GET()
                                .uri(URI.create(wireMock + "/joogle/inbox"), FuncDSL.use("joogle")),
                        FuncDSL.http("getEmailsFromJahoo").GET()
                                .uri(URI.create(wireMock + "/jahoo/inbox"), FuncDSL.use("jahoo"))))
        .build();
----
<1> Declare the secret groups for both clients together.
<2> `fork` runs the two authenticated calls in parallel; each picks up its own token.

== 8. Authenticate OpenAPI operations

If you use xref:http-openapi-tasks.adoc[OpenAPI tasks], supply the credentials through `oidc(...)` (or `oauth2(...)`):

[source,java]
----
FuncWorkflowBuilder.workflow()
        .use(u -> u.secrets("openapi"))
        .tasks(t -> t.openapi("imageService", f -> f
                .document(openApiDocumentUrl)
                .operation("listImages")
                .parameters(Map.of("Accept", "application/json"))
                .authentication(FuncDSL.oidc(baseUrl, CLIENT_CREDENTIALS,
                        "${ $secret.openapi.\"client-id\" }",
                        "${ $secret.openapi.\"client-secret\" }"))))
        .build();
----

== 9. Provide credentials

Credentials are supplied through xref:secrets.adoc[secrets] as shown in <<client-credentials-grant, section 3>>. In production, the same secret names resolve through any Quarkus https://quarkus.io/guides/credentials-provider[`CredentialsProvider`] (HashiCorp Vault, Kubernetes Secrets, …) without changing the workflow code. See xref:secrets.adoc[Resolve secrets securely] for details.

== 10. Override OIDC client configuration per workflow or task

The authentication declared in the Serverless Workflow DSL defines the _identity_ of the OIDC client — authority,
client ID, grant type, scopes, etc.
In many cases you also need to tune operational behavior (token expiry, refresh intervals,
custom headers) or override identity properties per environment (different auth server URL in production vs. development) without changing the workflow code.

Quarkus Flow lets you do this from `application.properties`, routed by workflow and task identity. Resolution follows this priority: the most specific match wins and is used entirely (no merging across levels):

1. **Task-level** (most specific) — `quarkus.flow.oidc.client."<namespace>\:<name>\:<version>\:<taskName>".<property>`
2. **Versioned workflow** — `quarkus.flow.oidc.client."<namespace>\:<name>\:<version>".<property>`
3. **Versionless workflow** — `quarkus.flow.oidc.client."<namespace>\:<name>".<property>`
4. **Named authentication** — `quarkus.flow.oidc.client.<authPolicyName>.<property>` — matches the name declared in `use(...)`, e.g. `use("keycloak")`
5. **Global defaults** (least specific) — `quarkus.flow.oidc.<property>`
6. **SDK fallback** — no override; the DSL-declared values are used as-is

The first level that has an entry in the configuration map is used; properties not set at that level fall through to the DSL-declared values (not to a less-specific configuration level).

[IMPORTANT]
====
Colons in the composite key must be escaped as `\:` in `application.properties` files. For example, for a workflow with namespace `myns` and name `my-workflow`, use `"myns\:my-workflow"`.
====

=== Example

[source,properties]
.application.properties
----
# Global defaults — apply to all OIDC clients created by Flow
quarkus.flow.oidc.access-token-expires-in=300S
quarkus.flow.oidc.refresh-token-time-skew=5S
quarkus.flow.oidc.headers.X-Tenant=default

# Named auth override — matches use("keycloak-prod") declarations
quarkus.flow.oidc.client.keycloak-prod.auth-server-url=https://keycloak.example.com
quarkus.flow.oidc.client.keycloak-prod.access-token-expires-in=600S

# Versionless override — applies to all versions of this workflow
quarkus.flow.oidc.client."myns\:my-workflow".auth-server-url=https://prod-auth.example.com
quarkus.flow.oidc.client."myns\:my-workflow".access-token-expires-in=600S

# Versioned workflow override
quarkus.flow.oidc.client."myns\:my-workflow\:2.0.0".auth-server-url=https://v2-auth.example.com

# Workflow + task override (most specific)
quarkus.flow.oidc.client."myns\:my-workflow\:1.0.0\:callSlowApi".access-token-expires-in=900S
quarkus.flow.oidc.client."myns\:my-workflow\:1.0.0\:callSlowApi".headers.X-Custom=value
----

=== Overridable properties

Both _identity_ properties (override what the DSL declares) and _operational tuning_ properties are supported:

.Identity overrides
[cols="1,1,2"]
|===
| Property | Type | Description

| `auth-server-url` | `String` | Base URL of the authorization server. Overrides the DSL `authority`.
| `token-path` | `String` | Relative path or absolute URL of the token endpoint.
| `discovery-enabled` | `Boolean` | Whether to use OIDC discovery (default: `false`).
| `client-id` | `String` | Client ID. Overrides the DSL `clientId`.
| `client-secret` | `String` | Client secret. Overrides the DSL `clientSecret`.
| `client-secret-method` | `String` | Method to send the secret: `BASIC`, `POST`, `POST_JWT`.
| `grant-type` | `String` | OAuth2 grant: `authorization_code`, `client_credentials`, `password`, `refresh_token`, `urn:ietf:params:oauth:grant-type:token-exchange`.
| `scopes` | `List<String>` | OAuth2 scopes. Overrides the DSL scopes.
| `audience` | `List<String>` | Target audiences. Overrides the DSL audiences.
|===

.Operational tuning
[cols="1,1,2"]
|===
| Property | Type | Description

| `access-token-expires-in` | `Duration` | Explicit expiry when the server response does not include `expires_in`.
| `access-token-expiry-skew` | `Duration` | Safety margin before the access token's actual expiry.
| `refresh-token-time-skew` | `Duration` | When to consider the refresh token expired.
| `absolute-expires-in` | `Boolean` | Treat `expires_in` as an absolute timestamp rather than a duration.
| `early-tokens-acquisition` | `Boolean` | Acquire the token eagerly at client creation time.
| `headers` | `Map<String,String>` | Custom HTTP headers for the token request.
| `refresh-interval` | `Duration` | Interval for asynchronous background token refresh.
|===

Headers merge additively across layers — a task-level header does not remove a global-level header with a different name.
